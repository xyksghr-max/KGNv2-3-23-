# KGNv2 -> KGN-Pro 方向改进计划

## 1. 当前项目判断

- 主干仓库：`/home/xyk/KGN-main`
- 参考仓库：`/home/xyk/KGN-main/KGN-Pro-main`
- 主开发路线：以 `KGNv2` 为主干，选择性吸收 `KGN-Pro` 中少数真正有价值、且确实接上训练链路的模块或思想
- 不采用的主线：不直接把 `KGN-Pro-main` 当作后续主开发仓库，也不以“补完整个 KGN-Pro 仓库”为当前毕设主线

原因：

- `KGNv2` 主链路完整，训练、测试、评估、数据生成都已打通
- `KGN-Pro` 仓库中存在部分真实新增训练侧代码，但主测试/主推理链路仍大量沿用 `KGNv2` 旧逻辑
- 直接补完整个 `KGN-Pro` 风险过高，不适合作为本科毕设主闭环路线

## 2. 总体目标

在保持 `KGNv2` 可训练、可测试、可评估闭环不破坏的前提下，分阶段引入 `KGN-Pro` 的有效思想，形成一个：

- 可运行
- 可训练
- 可评估
- 可解释
- 可写入毕业论文

的渐进式改进版本。

## 3. 版本控制与提交策略

### 3.1 基本规则

- 所有改动都只在 `KGNv2` 主仓库中进行
- `KGN-Pro-main/` 仅作为参考，不作为提交内容
- 每次只做一个功能改进，不把多个改动混在一次提交中
- 每个功能改进都必须先验证，再提交，再推送
- 下一次改动必须从“当前最新稳定分支”创建新分支

### 3.2 建议分支命名

- `feat/p0-baseline-regression`
- `feat/p1-error-analysis`
- `feat/p2-confidence-head`
- `feat/p3-confidence-ranking`
- `feat/p4-prob-pose-loss`
- `feat/p5-ablation-and-docs`

### 3.3 单功能迭代流程

1. 切到最新稳定分支并同步
2. 新建一个功能分支
3. 只实现一个功能
4. 做该功能对应的最小验证
5. 记录验证结果
6. 提交 commit
7. 推送到 GitHub
8. 合并后，再从最新稳定分支创建下一个功能分支

## 4. 统一验证基线

后续每个功能模块至少要通过与本模块相关的验证。推荐验证分成四层：

### 4.1 结构验证

- 配置可以正常解析
- 模型可以正常构建
- 数据集可以正常取样
- forward 不报错
- loss 不报错

### 4.2 小规模训练验证

- 在 `conda activate kgnv2` 环境下跑通小批量训练
- 至少确认 loss 正常下降或数值稳定
- 不允许出现 `nan`、维度错误、空 batch、显存异常增长

### 4.3 推理验证

- demo 可运行
- test 可运行
- 能正常输出 grasp candidates
- 不破坏现有 `decode -> PnP -> eval` 主链路

### 4.4 评估验证

- 至少完成一次已有评估脚本验证
- 若修改涉及排序、后处理或姿态恢复，必须额外比较修改前后的评估结果
- 若修改涉及训练损失，必须比较训练日志和测试指标

## 5. 改进优先级总览

| 优先级 | 模块 | 目标 | 风险 | 是否建议纳入毕设主线 |
| --- | --- | --- | --- | --- |
| P0 | 基线回归与实验护栏 | 固定当前可运行基线，补齐验证与日志输出 | 低 | 是 |
| P1 | 误差分析与中间结果可视化 | 明确 keypoint、scale、PnP 各环节误差来源 | 低 | 是 |
| P2 | 轻量 confidence/uncertainty 分支 | 引入 `KGN-Pro` 中最容易落地的思想 | 中 | 是 |
| P3 | confidence-aware 排序/过滤 | 让 confidence 真正影响推理和后处理 | 中 | 是 |
| P4 | 训练侧 probabilistic pose regularizer 原型 | 借鉴论文3训练思想，但不替换主推理链 | 中高 | 可选增强 |
| P5 | 消融实验、论文材料整理 | 为毕业论文形成闭环证据 | 低 | 是 |

## 6. 分模块实施计划

## 模块 P0：基线回归与实验护栏

### 6.1 模块作用

在任何算法改动前，先固化当前 `KGNv2` 的可运行基线，避免后续改动把训练、测试、评估链路破坏后无法快速定位。

### 6.2 对应论文

- 对应论文2 `KGNv2` 的完整工程基线
- 不直接引入论文3方法

### 6.3 计划改动

- 补一份基线回归文档
- 固定一组最小验证命令
- 补充必要的日志输出：
  - 模型 heads
  - 数据样本张量 shape
  - decode 输出 shape
  - PnP 成功率
- 补一份“功能改动前后统一验证清单”

### 6.4 重点文件

- `src/main.py`
- `src/test.py`
- `src/keypoint_graspnet.py`
- `src/lib/datasets/sample/grasp_pose.py`
- `src/lib/datasets/dataset/ps_grasp.py`
- `README.md`
- `docs/`

### 6.5 验证方法

- 跑通现有 demo
- 跑通现有 test
- 跑通一次小批量训练
- 保留基线日志和指标结果

### 6.6 通过标准

- 当前 KGNv2 全链路结果不回退
- 后续改动都可以用同一组基线命令复测

## 模块 P1：误差分析与中间结果可视化

### 6.1 模块作用

把 `KGNv2` 的关键误差拆开看清楚，先建立“改什么最值得”的证据链。

### 6.2 对应论文

- 论文2重点问题：`scale` 与 `pose` 分离
- 论文3切入点：关键点与 3D pose 之间监督不够强

### 6.3 计划改动

- 在测试阶段记录：
  - center score
  - predicted scale
  - reprojection error
  - PnP 成功/失败
  - 预测 grasp 数量
- 增加中间结果保存：
  - 2D keypoints 可视化
  - scale 分布
  - reprojection error 统计
  - 不同阈值下保留的 grasp 数量
- 若可行，增加 GT 对齐后的误差统计：
  - 2D keypoint 误差
  - translation 误差
  - rotation 误差

### 6.4 重点文件

- `src/test.py`
- `src/keypoint_graspnet.py`
- `src/lib/datasets/dataset/ps_grasp.py`
- `src/lib/utils/`

### 6.5 验证方法

- 对单物体和多物体各跑一组测试
- 成功生成误差统计文件与可视化结果
- 不改变原有评估指标输出

### 6.6 通过标准

- 能明确判断当前系统主要误差来自：
  - keypoint 本身
  - scale 分支
  - PnP 恢复
  - 后处理阈值

## 模块 P2：轻量 confidence / uncertainty 分支

### 6.1 模块作用

在不推翻 `KGNv2` 主链路的前提下，吸收论文3“confidence/uncertainty 建模”的低风险部分。

### 6.2 对应论文

- 论文3新增思想：`confidence map`
- 这里只做轻量 MVP，不直接照搬完整 probabilistic PnP

### 6.3 计划改动

- 在 `KGNv2` 中增加一个轻量 confidence head
- 优先做候选级或关键点级 confidence，而不是一开始就做完整分布建模
- 初版建议：
  - 先从 `center` 或 `candidate` 级 confidence 做起
  - 再决定是否扩展到 keypoint 级 confidence
- confidence 先作为辅助输出和分析信号，不立即强耦合进所有推理模块

### 6.4 重点文件

- `src/lib/opts.py`
- `src/lib/models/networks/pose_dla_dcn.py`
- `src/lib/models/model.py`
- `src/lib/trains/grasp_pose.py`
- `src/lib/datasets/sample/grasp_pose.py`

### 6.5 验证方法

- heads 正常构建
- forward 正常
- loss 正常
- 小批量训练不报错
- confidence 输出可视化正常

### 6.6 通过标准

- confidence 分支成功接入训练链路
- 不破坏原始 `KGNv2` 的 center/width/scale 主任务

## 模块 P3：confidence-aware 排序与后处理

### 6.1 模块作用

让新引入的 confidence 不只是“看起来有”，而是真正参与推理阶段的候选筛选和排序。

### 6.2 对应论文

- 论文3中 confidence 与 reprojection likelihood 共同作用于 pose 质量判断
- 这里先做工程可落地版本：confidence-aware ranking/filtering

### 6.3 计划改动

- 在推理端加入基于 confidence 的排序或过滤
- 与已有的 `score`、`reprojection error` 联合使用
- 初版建议组合方式：
  - `center score`
  - `confidence`
  - `reprojection error`
- 保持主恢复链路仍为 `decode -> cvIPPE -> scale recover`

### 6.4 重点文件

- `src/keypoint_graspnet.py`
- `src/lib/models/decode.py`
- `src/lib/detectors/grasp_pose.py`
- `src/lib/utils/post_process.py`

### 6.5 验证方法

- demo 可运行
- test 可运行
- 候选 grasp 数量和排序可记录
- 对比改动前后评估指标

### 6.6 通过标准

- 至少在一组测试中表现不差于基线
- 若总体指标提升有限，至少要能证明排序质量或误检控制更稳定

## 模块 P4：训练侧 probabilistic pose regularizer 原型

### 6.1 模块作用

吸收论文3中最有研究价值的部分，但只以“训练侧正则/辅助损失”的形式接入，不替换当前稳定推理主链。

### 6.2 对应论文

- 论文3核心思想：`probabilistic 2D-3D correspondence learning`
- 论文3关键模块：`probabilistic PnP`、`confidence weighting`、`pose distribution supervision`

### 6.3 实施策略

- 不直接复制 `KGN-Pro-main` 的整套实现
- 只迁移其中真正接上训练链路的有价值部分
- 并重新整理为适配 `KGNv2` 的版本
- 重点避免以下问题：
  - 硬编码相机内参
  - 固定图像中心/分辨率
  - 固定 batch 形状
  - 训练有、测试不用

### 6.4 初版目标

- 把 probabilistic pose loss 做成可开关的辅助项
- 默认不影响原有确定性推理链
- 可以做消融：
  - baseline
  - baseline + confidence
  - baseline + confidence + pose regularizer

### 6.5 重点文件

- `src/lib/trains/base_trainer.py`
- `src/lib/trains/grasp_pose.py`
- `src/lib/models/`
- `src/lib/datasets/sample/grasp_pose.py`
- `src/lib/utils/post_process.py`

### 6.6 验证方法

- 小批量训练可运行
- loss 数值稳定，无 `nan`
- 不引入固定尺寸/固定 batch 假设
- 关闭该 loss 时，系统行为应退化回普通 `KGNv2`

### 6.7 通过标准

- 可作为一个独立开关模块存在
- 至少完成训练侧原型验证
- 若验证成本过高，可降级为“论文讨论 + 局部实验”

## 8. 明确不作为当前主线的内容

以下内容暂不作为主线目标：

- 直接把 `KGN-Pro-main` 改造成完整论文3项目
- 直接用 `KGN-Pro-main` 替代当前 `KGNv2` 主仓库
- 在当前阶段重写整套推理链为 full probabilistic PnP inference
- 为了追求“更像论文3”而破坏当前已经打通的训练/测试/评估闭环

## 9. 每次功能提交前必须回答的四个问题

1. 这次改动只解决一个清晰问题了吗？
2. 这次改动有对应验证吗？
3. 这次改动是否破坏了当前 `KGNv2` 主链路？
4. 这次改动是否能清楚写进毕业论文？

如果其中任一项答案是否定的，则不应直接提交。

## 10. 当前建议的第一个功能改进

建议第一个正式功能分支从 `P0 + P1` 开始，优先做：

- 基线回归护栏
- 中间结果记录
- 误差统计可视化

原因：

- 风险最低
- 能最快建立“为什么改、先改哪里”的证据链
- 能直接为后续 `confidence` 模块提供设计依据
- 能直接产出论文中可用的分析图和误差表
