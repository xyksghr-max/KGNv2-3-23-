## 4. 论文3相对论文2的有效新增思想梳理

下表只讨论“论文3相对论文2新增了什么”，并且明确区分：

- 论文方法是否成立
- 对当前项目是否有工程价值
- 当前 `KGN-main` 是否已实现
- `KGN-Pro-main` 是否真的提供了可借鉴代码

| 论文3新增点 | 相对论文2的新增内容 | 理论价值 | 当前 `KGN-main` 状态 | `KGN-Pro-main` 状态 | 新阶段优先级 |
| --- | --- | --- | --- | --- | --- |
| `confidence-aware 2D-3D correspondence weighting` | 不再默认所有 2D keypoint correspondence 同等可靠 | 高 | 已有轻量 `conf_branch` 与 `conf_fusion`，但还不是论文3式 correspondence weighting | 有局部影子，主要体现在 `w2d` 与 trainer-side weighting | 高 |
| `probabilistic PnP / EPro-PnP` | 从确定性 PnP 恢复扩展到概率式姿态估计 | 高 | 未实现 | 训练侧局部接入，但未接管主推理链 | 中 |
| `3D pose distribution supervision` | 不再只看 2D 回归误差，而是通过姿态分布做训练侧约束 | 高 | 未实现 | 有 `MonteCarloPoseLoss` 与 trainer-side `compute_loss` | 中 |
| `nearest-neighbor matching for multi-grasp supervision` | 显式处理“一物多抓取”的监督分配 | 中高 | 未实现 | 只有不够干净的固定长度 `grasp_pose` GT 打包原型 | 中 |
| `end-to-end gradient through pose recovery` | 让 3D 姿态监督回传到图像编码器 | 高 | 未实现 | 训练侧部分存在，但实现较原型化 | 中 |
| `replace deterministic inference with probabilistic inference` | 用概率姿态恢复替代原有主推理链 | 理论上有价值 | 未实现 | 未真正落到主测试/主推理链 | 低 |

### 4.1 当前最值得迁移的思想

最值得优先迁移到 `KGN-main` 的，不是“整套论文3”，而是以下三类：

1. `confidence / correspondence weighting`  
   这是对你当前 `P2 / P3` 的自然升级，风险最低，也最容易形成论文叙事闭环。

2. `clean training-side probabilistic pose loss prototype`  
   可以作为论文3训练思想的“干净、可开关、可消融”版本，而不是直接照搬 `KGN-Pro-main` 的原型代码。

3. `multi-grasp target matching / supervision cleanup`  
   这部分有助于把论文3说的“抓取分布监督”写得更扎实，但应该在更低风险模块稳定后再推进。

### 4.2 当前不应优先追求的内容

- 不优先追求“把测试 / demo / 推理主链全部换成 probabilistic PnP”
- 不优先追求“让 `KGN-main` 完全长成 `KGN-Pro-main` 的样子”
- 不优先做：
  - 硬搬整套 `KGN-Pro-main`
  - 复杂但难验证的结构重写
  - 没有清晰消融路线的大改动

## 模块 T3：Clean Probabilistic Pose Loss 原型（更新版）

### 10.1 Summary

`T3` 的目标不再是“尽量像 `KGN-Pro-main`”，而是做一个训练侧、可开关、可消融、无硬编码、与当前主推理链解耦的概率姿态辅助损失原型。

它只迁移论文3里最有研究价值的部分：
- `probabilistic pose supervision（概率姿态监督）`

本阶段明确不做：
- 新的推理主链
- `w2d/x2d` 头
- `per-keypoint confidence`
- `multi-grasp nearest-neighbor matching`

`T3.1` 的结论已经锁定：
- 第一版以 `GT` 正样本对齐方式接入训练；
- 不走 `top-K decode（top-K 解码候选）` 路线；
- 不修改 `test.py / keypoint_graspnet.py / decode.py` 主流程。

### 10.2 Key Changes

#### 10.2.1 T3.1 规格冻结

- `T3` 只做训练侧 `auxiliary pose loss（辅助姿态损失）`，不改当前 `decode -> PnP -> eval（解码 -> 位姿恢复 -> 评估）` 主链。
- 不直接复制 `KGN-Pro-main/src/lib/trains/base_trainer.py` 的 `compute_loss(...)`；`BaseTrainer（基础训练器）` 保持不动，所有新逻辑放在 `grasp_pose` 任务侧。
- 不直接复制 `KGN-Pro-main/src/lib/datasets/sample/grasp_pose.py` 的固定长度 `grasp_pose` 打包；`T4` 之前不做 `multi-grasp matching（多抓取匹配）`。
- 第一版不引入 `w2d/x2d`，不引入 `per-keypoint confidence（逐关键点置信度）`；`T3` 只隔离“概率姿态监督”本身，避免和 `T2` 的 `confidence` 语义混在一起。
- `3D keypoint model（3D 关键点模型）` 必须复用当前主仓已有的 `canonical grasp keypoints（规范抓取关键点）`，来源统一于现有 `grasp_kpts.py / BasePnPSolver.get_grasp_kpts_3d()`，禁止再硬编码一套新的四点模型。

#### 10.2.2 T3.2 数据与接口准备

- 在训练样本中新增最小必需 `GT` 字段，全部按 `max_grasps（每图最大抓取数）` 对齐，不引入固定 batch 大小：
  - `grasp_pose_cam`：`(max_grasps, 4, 4)`，对应当前标签顺序的 `GT` 抓取位姿，`camera frame（相机坐标系）` 下。
  - `grasp_pose_mask`：`(max_grasps,)`，标记哪些 `GT pose` 有效。
  - `camera_intrinsic`：`(3, 3)`，每张图像自己的相机内参。
  - `pose_loss_meta_c`：`(2,)`，当前样本的仿射变换中心 `c`。
  - `pose_loss_meta_s`：`(1,)` 或标量，当前样本的仿射变换尺度 `s`。
  - `pose_loss_img_hw`：`(2,)`，原图高宽。
  - `pose_loss_valid`：标记该样本是否允许参与姿态辅助损失。
- `pose_loss_valid` 的默认规则：
  - 样本若发生 `horizontal flip（水平翻转）`，则整张样本的 `pose_loss_valid = 0`。
  - 原因：翻转后的图像与原始 `camera pose（相机位姿）` 不再物理一致；第一版宁可跳过，也不引入错误监督。
- 随机裁剪与缩放保留，但训练侧 `pose loss` 必须把预测 `2D keypoints` 通过当前样本的 `c/s` 变换回原图坐标，再送入概率 `PnP`；禁止使用硬编码 `c=[320,240]`、`s=[672,512]`、`out_res=128x168` 之类常数。

#### 10.2.3 T3.3 干净的训练侧姿态辅助损失接线

- 新增最小开关接口：
  - `--prob_pose_loss`
  - `--prob_pose_weight`
- 第一版不暴露更多 solver 超参数给命令行；内部默认固定使用一套“低显存、低风险原型配置”，后续若 `T3.1` 云端验证通过，再逐项放宽并恢复更强配置。
- 依赖组织方式：
  - 把 `EPro-PnP` 的最小可运行 Python 依赖集 vendor 到 `third_party/epropnp/`，不复制 `tools/exps_cfg/`，不依赖外部 `YAML`。
  - 迁移并保留 `MonteCarloPoseLoss` 到主仓，例如 `src/lib/models/monte_carlo_pose_loss.py`。
  - 新增一个任务内辅助模块，例如 `src/lib/models/prob_pose_aux_loss.py`，专门负责：
    - 从当前输出和 batch 中取出 `GT` 正样本对应的预测 `2D keypoints`
    - 还原到原图坐标
    - 组织 `canonical 3D keypoints`
    - 调用 `EProPnP` 与 `MonteCarloPoseLoss`
- 关键接线策略：
  - `BaseTrainer` 不改。
  - 在 `GraspPoseLoss_clf.forward(...)` 内部，若 `--prob_pose_loss` 开启，则额外计算 `prob_pose_loss` 并加到总损失里。
  - 这条辅助损失只作用于 `reg / kpts_center_offset / shared backbone（中心偏移 / 关键点偏移 / 共享主干特征）`，不改测试端行为。
- 第一版 `correspondence weighting（对应关系加权）` 默认采用 `uniform weights（均匀权重）`，不复用 `T2` 的 candidate-level `conf`，也不引入新 `head`。
  - 理由：`T2` 已经负责 `confidence / ranking（置信度 / 排序）`；`T3` 先单独验证“概率姿态监督本身有没有价值”。

#### 10.2.4 T3.4 验证与日志

- 训练侧新增统计项：
  - `prob_pose_loss`
  - `prob_pose_valid_count`：当前 batch 中真正参与姿态辅助损失的 `GT grasp` 数量
  - `prob_pose_cost_mean`：姿态辅助监督里的平均 `target cost（目标代价）` 或等价稳定统计
  - `prob_pose_active_weight`：当前实际启用的概率姿态损失权重
- 关闭 `--prob_pose_loss` 时，系统必须完全退回 `T2` 当前行为。
- 当前阶段不直接进入正式 5 轮对照实验，先做：
  - `smoke train（烟雾测试）`
  - `stability probing（稳定性探测）`
  - 再在云服务器做正式验证

### 10.4 Test Plan

- 结构验证
  - 新增字段进入 `train batch` 后，不破坏当前 `dataloader / collate`。
  - `--prob_pose_loss` 关闭时，`loss_stats（损失统计）` 中不出现新项，训练行为与 `T2` 一致。
  - `--prob_pose_loss` 开启时，出现：
    - `prob_pose_loss`
    - `prob_pose_valid_count`
    - `prob_pose_cost_mean`
    - `prob_pose_active_weight`
- 小规模训练验证
  - 先跑 `1 epoch / 少量 iter` 的 `smoke train`：
    - 无 `nan`
    - 无 batch 维度错误
    - 无空样本崩溃
    - `prob_pose_valid_count` 在非翻转样本中大于 `0`
  - 明确检查：
    - `flip` 样本是否被正确跳过
    - 当前 `c/s/intrinsic` 是否已真正来自样本，而不是硬编码
- 正式对照实验
  - 与 `T2` 当前强主线做控制变量训练对比。
  - 测试保持：
    - `conf_fusion_alpha = 0.7`
    - `conf_fusion_min_conf = 0.3`
  - 主要比较：
    - `GSR（抓取成功率）`
    - `GCR（抓取覆盖率）`
    - `OSR（物体成功率）`
    - `images_with_any_eval_success（至少一条评估成功的图像数）`
    - `eval_successful_prediction_total（评估成功预测总数）`
    - `accepted_reproj_mean（最终接受候选平均重投影误差）`
- 结论判定
  - 若正式实验优于或接近 `T2`，则 `T3` 作为“训练侧概率姿态监督增强”成立。
  - 若指标不提升，但训练稳定、实现干净、归因清晰，则 `T3` 仍可作为论文中的“论文3训练思想迁移原型”成立。
  - 只有当 `T3` 训练稳定且显示出多抓取监督分配确实成为瓶颈时，才进入 `T4`。

### 10.5 Assumptions

- `T3` 第一版不做 `multi-grasp nearest-neighbor matching（多抓取最近邻匹配）`；这部分留给 `T4`。
- `T3` 第一版不引入 `w2d/x2d` 或 `per-keypoint confidence（逐关键点置信度）`；这部分留给未来条件性增强。
- `T3` 第一版不改任何测试、`demo`、主推理文件；所有收益都必须来自训练侧辅助监督。
- `T3` 的目的不是“完整复现论文3”，而是做一个适合当前主仓、适合本科毕业设计叙事的干净原型。
- 当前 `T3.1` 的正式验证场景已锁定为：
  - 本地 Codex 负责代码审计、必要修复、文档记录、commit 和 push
  - 云服务器负责 pull 最新分支、GPU 训练/测试、保存日志和实验结果
  - 云端单实验目录打包回传本地后，由本地 Codex 继续分析。
